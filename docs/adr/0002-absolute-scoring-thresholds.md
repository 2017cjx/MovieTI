# 0002: 軸判定はセッション内相対ではなく絶対基準を使う

## Status
Accepted (2026-07-14)

## Context
[[0001]]でプリセットを廃止し、質問選定エージェントがTMDbから都度映画を選ぶ方式にした結果、
「全ユーザー共通の固定映画セット」が存在しなくなった。当初のスコアリング設計（軸の判定に
提示された50本サンプルの中央値を使う）は、この固定セットの存在を前提にしていたため、
そのままでは成立しなくなった。

セッションごとに提示される映画が異なる中で、H/L・N/O・M/U・W/Fをどう判定するかで2案を検討した。

1. セッション内相対: そのユーザーに提示された映画群だけで中央値/閾値を計算する。
2. 絶対基準: TMDb全体のデータに基づく固定のしきい値を事前に決め、全ユーザー共通で使う。

案1は実装は簡単だが、エージェントが偶然メジャー作品ばかり選んだ場合とマイナー作品ばかり
選んだ場合とで、同じ回答内容でも判定結果が変わってしまう。診断結果の再現性・比較可能性が
失われ、「当たってる感」を損なうリスクがあると判断した。

## Decision
案2（絶対基準）を採用する。各軸の判定に使う基準は以下の通り（具体的な数値は
実装時に実データで調整する = 要チューニング）。

- 大衆性(M/U): TMDbの`vote_count`（popularityではなく、より安定した「累計でどれだけ
  多くの人に見られたか」の指標）に固定の閾値を設け、メジャー/マイナーを判定。
- 時代志向(N/O): 公開年に固定のアンカー（例: 直近15年以内かどうか）を設け判定。
  セッションごとの分布に依存しない。
- ジャンル幅(W/F): distinct数だけでなく割合も使う。高評価作品のうち、最も評価が
  集中しているジャンル上位1〜2個が占める割合を計算し、固定のカットオフ値で判定する。
  理由: distinct数だけだと鑑賞数が多い人ほど自然とジャンル数も増えてしまい、
  鑑賞量(H/L)と相関してしまう（軸同士が独立でなくなる）。割合にすることで鑑賞量に
  依存せず「本当に偏っているか」を測れる。
  **実装バグ修正（2026-07-15）**: 当初`src/scoring.ts`実装では分母を「高評価映画の本数」に
  していたが、1本の映画は複数ジャンル（2〜3個）を持つため分子（上位2ジャンルの
  合計出現数）が分母を上回りやすく、ほぼ常にF側に張り付く（軸が機能しない）バグが
  あった。分母をジャンルタグの総出現数に修正済み。
- 鑑賞量(H/L): 提示された質問数に対する「見た」回答の割合（絶対数ではなく%）で判定。
  質問総数が可変（早期終了などがある場合）でも成立するようにする。

## Consequences
- 各しきい値の具体的な数値はTMDbの実データ分布を見て事前にチューニングする作業が必要。
  質問選定エージェントの候補プールの傾向（人気作に寄っていないかなど)と合わせて検証する。
- ユーザー間で結果が比較可能になるため、[[レア度スコア]]（未解決）の実装の土台になる。

**鑑賞量(H/L)のしきい値修正（2026-07-15）**: 当初の実装は「見た」回答の割合が50%を
超えたらHeavyという、可能な範囲（0-100%）の中央値をそのまま閾値に使う設計だった。
実際にライブテストを繰り返すと、インジケーターが毎回ほぼ真ん中に寄ってしまう問題が
見つかった。原因は「50%」という閾値が母集団の実態と乖離していたこと。日本人の
映画館での年間鑑賞回数は平均1.4回程度であり、80問中50%（40本）もの映画を認識できる
時点で、一般的な母集団の中では既に極端な「めちゃくちゃ見てる人」の層に入る。可能な
範囲の中央値（50%）を「平均的な人」の基準点として使うのは、実際の分布が大きく偏って
いる場合には誤り——年齢分布が20代に偏った集団で「40歳以上か未満か」を平均の境界線に
するようなもので、実態を反映しない。`VOLUME_RATIO_THRESHOLD: 0.5`を廃止し、
`VOLUME_NEUTRAL_RATIO: 0.15`（中立点を15%に下げる）と`VOLUME_SPREAD: 0.35`
（この幅で±1に到達）に置き換えた。具体的な数値はどちらも実データではなく推定値であり、
SPEC.md 9節の「実データで調整」対象のまま。
