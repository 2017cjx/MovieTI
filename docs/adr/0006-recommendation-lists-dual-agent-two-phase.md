# 0006: 結果画面のおすすめ2リストは、リストごとに独立した1フェーズAIエージェント＋独立エンドポイントで生成する

## Status
Accepted (2026-07-17)

## Context
結果画面に新機能として「あなたへのおすすめ」（"You might like these movies"）と
「視野を広げる映画」（"Movies that could broaden your horizon"）の2リストを追加する
（`/grill-with-docs`セッションで設計、詳細は`CONTEXT.md`「おすすめ2リスト（結果画面）」）。
これまでこのアプリに映画レコメンド機能は存在せず（映画が登場するのは診断中の質問と、
結果画面のお気に入り行=ユーザー自身の高評価作品のみ）、ゼロから設計する必要があった。

設計時、以下の点でシンプルな案とAI駆動の案を比較した。

- **選定ロジック**: TMDbの`discover`/`recommendations`パラメータをデターミニスティックに
  組み立てる案（[[軸 (Axis)]]の一致/不一致だけで機械的に判定）と、LLMエージェントに
  委ねる案。
- **エージェント構成**: 既存の[[質問選定エージェント]]・仮説形成エージェントは
  「裏方・構造化出力のみ・キャラクター性なし」という役割分担
  （`CONTEXT.md`「エージェントの役割分担」）を厳守してきた。この新機能でも同じ
  節度を守るか、より踏み込んだAI活用にするかが論点だった。
- **呼び出し回数**: 既存2エージェントは「往復待ちを増やさない」ことを一貫して優先
  してきた（20問目チェックポイントを独立エンドポイントにせず`/api/next-batch`に
  吸収した設計判断が象徴的）。この新機能でも同じ節約志向を適用するかどうか。

ユーザーは一貫して、より踏み込んだAI活用側（レコメンド方式は「あなたへのおすすめ」を
TMDbの`recommendations`類似度、「視野を広げる映画」を4軸すべての逆張り、選定自体を
エージェント駆動、しかもリストごとに完全独立した2エージェント）を選択した。

**2026-07-17追記（同日改訂）**: 当初案は各エージェントを2フェーズ（フェーズ1で検索
条件決定、フェーズ2で候補プールから最終5本をAIが再ランキング）とし、結果画面全体で
最大4回のLLM呼び出しになる設計だった。ユーザーから「2 agents for each list is a bit
unnecessary」という指摘を受け、各エージェントを1フェーズ（検索条件決定のみ、
[[質問選定エージェント]]と同型）に縮小し、呼び出し回数を最大2回に半減した。
同じタイミングで、両リストを1本のエンドポイント`POST /api/recommendations`に
まとめる当初案も再検討した。1つのHTTPレスポンスは両パイプラインの完了を待たないと
送れないため、「リストごとに独立したローディングスピナーを出す」というUI決定と
矛盾することが判明し、エンドポイントも2リストで分離することにした
（ユーザー指摘: 「API points should be twofolds right? for each list」）。

## Decision
1. **2つの独立したエージェント**を新設する（1エージェントが両リストを兼務する案は
   不採用）。役割分担は保つが、「呼び出し回数を1つでも減らす」という既存の節約志向
   よりも「リストごとに完全に独立して失敗・調整できること」を優先する。
2. 各エージェントは**1フェーズのみ**で動く: エージェントがTMDb検索条件（種映画ID/
   `discover`条件）をJSON出力 → デターミニスティックなコードがTMDbを実行（既存
   ガードレール適用、8〜10件取得）→ AIによる再ランキングなしで、シャッフル済み
   候補の先頭5件を採用する。LLMは一貫してTMDbを直接叩かない（既存の「LLMの判断と
   API呼び出しは別レイヤー」原則を維持）。
3. 結果画面で**最大2回のLLM呼び出し**（2リスト×1フェーズ）が発生する。2リストの
   パイプラインは互いに独立なため並行実行する。
4. モデルは既存の[[質問選定エージェント]]・仮説形成エージェントと同じ
   `llama-3.2-3b-instruct`を2呼び出しとも使う（構造化出力＝小型モデルという
   既存の役割分担を維持、[[ライブLLM演出]]の大型モデルとは別枠）。
5. 失敗時は[[フォールバックプール]]のような代替表示をせず、**そのリストのセクション
   ごと省略する**。2リストは独立に失敗しうる。
6. **2つの独立したAPIエンドポイント**、`POST /api/recommend-similar`と
   `POST /api/recommend-horizon`を新設する（既存の`/api/next-batch`・
   `/api/flourish`の2エンドポイント構成から4つに増える。[[ライブLLM演出]]の
   `/api/flourish`は「決まったデータから文章を書く」役割に特化しているため、
   無関係の新規AI選定処理をそこに混ぜるのは責務違反と判断し、統合案は不採用。
   1本の`/api/recommendations`に両リストをまとめる案も、独立したローディング
   スピナーの実現と両立しないため不採用）。フロントエンドはこの2エンドポイントに
   独立した2本の`fetch`を発行する。
7. スクリーニングフェーズ（Q1-20、固定約50本の`fallback_pool.json`が出典）由来の
   評価データは、深掘りフェーズ（Q21-80、TMDbランダム選出）由来のデータより
   **優先度を下げる**（除外ではない）。「あなたへのおすすめ」のシード映画選定、
   「視野を広げる映画」の`genre_coverage`/`language_coverage`集計、いずれも
   深掘り由来データを優先し、不足時のみスクリーニング由来で補う。理由:
   スクリーニングは全セッション共通の小さい固定プールから出題されるため、これを
   そのまま種にすると複数回の診断・複数ユーザー間でリストが似通ってしまう
   （ユーザー要望: 「each time of taking quiz, I wanna show diff set of
   recommended movies' list」）。
8. 上記7の結果、「あなたへのおすすめ」のシード候補（最大5本）が完全に
   デターミニスティックに決まるため、「エージェントがシード映画IDを決める」という
   当初の記述は矛盾になっていた（決めるものがない）。解消: エージェントの実際の
   仕事を「デターミニスティックに絞り込まれた最大5本の中から、`tasteHypothesis`を
   手がかりにTMDb `/recommendations`へ実際に投げる1〜2本を選ぶ」に再定義する
   （5本全部に個別に`/recommendations`を叩いてマージする案、AI呼び出し自体を
   省略する案は不採用）。TMDb呼び出しを最大5回から1〜2回に抑えつつ、
   エージェントに実質的な判断（「どの高評価作品が全体的な好みを一番代表するか」）を
   残す。
9. 2リストの取得結果は`localStorage`にキャッシュする（`useQuizState.ts`の
   `STORAGE_KEY`に相乗り、リスタート時に消える）。[[ライブLLM演出]]のflourish
   テキストは無キャッシュで毎回再取得する既存パターンだが、そのまま踏襲すると
   TMDb候補選定のランダム性によりページリロードのたびにおすすめ内容が変わり、
   「同じ診断結果を見ているのに表示が変わる」というバグに見える挙動になるため、
   この新機能ではあえて既存パターンを踏襲しない。

## Consequences
- 結果画面のLLM呼び出しが1回（既存の[[ライブLLM演出]]）から最大3回
  （既存1回＋新規最大2回、+仮説形成含めると診断全体ではさらに多い）に増える。
  レイテンシ・失敗率のモニタリングが今まで以上に重要になる
  （`docs/validation-runs/`のE2E検証の対象を広げる必要あり）。
- プロンプトファイルが3つ（question/hypothesis/result-writer）から5つに増える
  （`prompts/recommend-similar-agent.md` / `prompts/recommend-horizon-agent.md`）。
  `scripts/generate-prompts.mjs`のビルドパイプラインはそのまま流用できる。
- APIエンドポイントが2つから4つに増え、`src/api-types.ts`の「最小2エンドポイント」
  という設計方針（HTTP APIの契約セクション参照）を明示的に破る初めてのケースになる。
  将来さらにエンドポイントが増える場合、この判断が前例として参照される可能性がある。
- 各リストが独立に失敗しうる（最大パターンとして、両方失敗して結果画面に新規セクションが
  一切表示されないケースも起こり得る）。これは意図的な仕様（フォールバックプールを
  使った代替表示より「AIが実際に考えた結果でなければ出さない」ことを優先した）だが、
  デモ本番でその状況が起きた場合の見え方は未検証。
- フェーズ2（AI再ランキング）を廃止したことで、TMDbが返した8〜10件のうち先頭5件を
  機械的に採用する形になり、[[質問選定エージェント]]の`discoverMovies`が既に持つ
  「ランダムページ選択＋シャッフル」ロジックの再利用で足りる。新規の候補ランキング
  ロジックを実装・検証する必要がなくなった。
