# 0001: 質問選定はプリセットを持たず、TMDbを叩くライブエージェントにする（スクリーニングは2026-07-15に部分的に撤回）

## Status
Accepted (2026-07-14)、2026-07-15にスクリーニングフェーズのみ部分的に改訂（下記「2026-07-15の改訂」参照）

## Context
診断の50問（前後）で見せる映画をどう選ぶかについて、2つの案を検討した。

1. 事前に80〜100本程度の映画を人手/AIでキュレーションし、固定プリセットとして同梱する。
   スコアリングの重み付けや軸の分散を design-time に検証できるため、デモの安定性・再現性が高い。
2. 固定プリセットを持たず、診断中にLLMエージェントがTMDbの`discover` APIを都度叩いて
   カタログ全体から次の映画を選ぶ。ユーザーごとに実際に問われる映画が変わる。

案1は安全だが、「膨大な映画情報の中から本当に一人一人向けに掘り下げていく」という
プロダクトの核（インスタに上げたくなるほど"刺さる"個別性）に対して弱いと判断した。
固定プリセットに絞った時点で、結局は静的なアンケートに近くなってしまう。

一方で案2をそのまま素朴に実装すると、以下の具体的なリスクがある。

- デモ中にTMDb APIが詰まる/落ちると、次の映画が出てこず診断が止まる。
- 1問ごとにLLMを呼ぶと、50問で合計1〜2分半の「考え中」待ちが発生し、
  「テンポよく答える」という体験を損なう。

## Decision
案2（プリセットなし・ライブTMDb・エージェント方式）を採用する。ただし2つの緩和策を組み合わせる。

- **バッチ化**: LLMは1問ごとではなく、5問程度ごとに「次のバッチで狙う軸と検索条件」を
  まとめて決定する。TMDb呼び出し・LLM呼び出しの往復回数を50回から10回程度に減らす。
- **フォールバックプール**: TMDbまたはLLMの呼び出しが失敗/タイムアウトした場合にのみ、
  ローカルに保持した少数（10〜20本）の映画キャッシュに自動的に切り替える。
  通常フローでは一切使われない保険。

## Consequences
- 質問の並びはユーザーごとに毎回変わり、design-time に全パターンを検証することはできない。
  合成ペルソナによる検証（4節）は「軸の重み付けロジック」の妥当性検証には使えるが、
  「エージェントが選ぶ映画の並び」自体の当日の挙動は保証できない。本番前のリハーサルを
  複数回行い、明らかにおかしな挙動（同じ映画の重複、メタデータ欠損の映画が出るなど）が
  ないか目視確認する運用でカバーする。
- TMDbの`discover`エンドポイントに渡すクエリパラメータ（ジャンルID、年代範囲、
  vote_count下限など）はLLMが自由文で組み立てるのではなく、決まった型のツール
  （関数呼び出し）として与え、パラメータのバリデーション（例: vote_count.gte を
  一定値以上にしてポスター/メタデータ欠損作品を除外）を行う。
- フォールバックプールの選定・更新は別途必要（要検討事項として残す）。

## 2026-07-15の改訂: スクリーニング（Q1-20）はプリセット、深掘り（Q21-80）は引き続きライブ

実際にライブ実装して動かしたところ、スクリーニングフェーズの最初の呼び出しは
全軸スコアが0（何も分かっていない状態）で毎回まったく同じ入力になるため、
エージェント（LLM）が毎回似たような検索条件（人気順ソートなど）に収束し、
「何回試しても最初の数問が同じような大作ばかりになる」という体感上の問題が
ユーザーテストで判明した。

この期間に`data/fallback_pool.json`を大幅拡充（ジャンル・原語・監督・年代を
意図的に分散させた50本規模）していたこともあり、方針を改める。

**改訂した決定**: スクリーニング（Q1-20）は`fallback_pool.json`からランダム抽出する
方式に変更し、深掘り（Q21-80）は本ADRの当初方針（ライブTMDb・エージェント方式）を
維持する。

理由:
- スクリーニングの目的は「4軸それぞれの両極端をまんべんなく聞き、暫定スコアをつける」
  こと（CONTEXT.md「質問選定エージェント」）であり、必ずしもライブ検索でなくても
  達成できる。手動でジャンル・原語・監督・年代を分散させたプールの方が、
  コールドスタート時のエージェントより確実に多様性を保証できる。
- 深掘り（全体の75%、60問）は引き続きライブ検索なので、「膨大な映画情報の中から
  一人一人向けに掘り下げていく」という当初の製品の核（本ADR冒頭の懸念）は
  大部分維持される。
- 副次的な利点: スクリーニング中はLLM/TMDb呼び出しが一切不要になり、
  レイテンシ・コストの両方が下がる。

**Consequencesの更新**: フォールバックプールは「障害時のみの保険」という
当初の位置づけから、「スクリーニングフェーズの主要な供給源」へと役割が広がった。
`NextBatchResponse.source`に`"preset"`（スクリーニングでの意図的な使用）を追加し、
`"fallback"`（深掘りでの障害時使用）と区別できるようにした
（`src/api-types.ts`）。プールの選定・更新はもはや「後回しでよい要検討事項」ではなく、
診断の最初の20問の体験を直接左右する重要な作業になった。
